# 日历功能扩展实现计划

## 需求分析

### 需求1：长期事项标注提示功能

#### 原始需求
- 将日历显示从5周×7天改为4周×7天
- 原第五周位置改为"长期提醒事项"区域
- 支持增删改长期事项
- 不与飞书日历同步
- 如果占位过多，先设置3格，剩余占位用于下次更新

#### 可行性分析
**完全可行**。当前项目使用 React + localStorage 架构，实现此功能的技术路径清晰：

1. **布局调整**：修改 `getCalendarDays()` 函数，从生成35天（5周）改为28天（4周）
2. **数据存储**：长期事项可存储在 localStorage 中（与现有日程存储方式一致）
3. **UI组件**：新增 `LongTermReminders` 组件，支持列表展示和增删改操作
4. **占位设计**：采用灵活的栅格布局，3格用于长期事项，4格保留扩展

#### 优化建议

**方案A：底部固定区域（推荐）**
- 日历区域改为4周×7天（28格）
- 底部新增独立区域"长期提醒事项"，横向占满整行
- 采用卡片式布局，每个事项一个卡片
- 初始显示3-5个事项卡片，可通过滚动查看更多
- 优点：布局清晰，不影响日历主体，扩展性好

**方案B：替代第五周区域**
- 原第五周的7个格子中，左侧3-4格显示长期事项
- 右侧保留为日历格子（用于显示跨月日期）
- 优点：空间利用率高
- 缺点：布局较为复杂，视觉不够统一

**推荐采用方案A**，理由：
- 视觉上更清晰，长期事项与日历日期区分明确
- 不影响日历的完整显示（4周可以完整显示当月日期范围）
- 扩展性更好，未来可以增加更多长期事项

### 需求2：通讯录功能

#### 原始需求
- 放在右上角（飞书日历管理中心、导入/导出按钮旁）
- 点击后允许增减通讯录内容
- 只需要"姓名"和"电话"两个字段

#### 可行性分析
**完全可行**。实现路径：

1. **入口按钮**：在顶部导航栏新增"📇"按钮
2. **数据存储**：存储在 localStorage 中
3. **UI组件**：新建 `ContactsModal` 组件，支持列表展示和增删改
4. **数据格式**：`{ id: string, name: string, phone: string }`

#### 优化建议

1. **搜索功能**：通讯录数量增多后，建议增加搜索/过滤功能
2. **批量操作**：支持批量导入/导出（如 CSV 格式）
3. **快捷拨号**：虽然是桌面应用，但可以考虑"复制到剪贴板"功能
4. **分类标签**：未来可增加"家人"、"同事"等分类标签

```typescript
interface LongTermReminder {

  description?: string
```

#### 1.2 存储方案
```typescript
// 存储在 localStorage
```

## 风险评估

| 风险项 | 影响 | 概率 | 应对措施 |
|--------|------|------|----------|
| 4周日历无法完整显示月份 | 中 | 低 | 4周=28天，配合月份切换可覆盖所有日期 |
| 长期事项占用空间过多 | 低 | 中 | 采用固定高度+滚动方案 |
| localStorage 容量限制 | 低 | 极低 | 通讯录和长期事项数据量很小 |

- ✅ 所有新增功能数据持久化
