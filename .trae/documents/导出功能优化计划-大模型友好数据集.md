## 待确认问题及建议

### 问题 1: 路径在文字中间的情况

**场景**:
```
讨论 C:\docs\contract.docx 这个文件的内容
```

**识别结果**:
```json
{
  "description": "讨论 这个文件的内容",
  "attachments": [{ "path": "C:\\docs\\contract.docx", "name": "contract.docx" }]
}
```

**分析**:
- ✅ **优点**: 文字和附件分离清晰
- ⚠️ **缺点**: 文字读起来不连贯（"讨论 这个文件的内容"）

**建议方案**:

**方案 A: 保持现状（推荐）**
- 识别为：`description: "讨论 这个文件的内容"`
- 理由：
  - 大模型可以根据附件路径理解上下文
  - 实际使用中这种情况较少
  - 保持解析逻辑简单
  - `rawDescription` 保留完整原文

**方案 B: 保留路径标记**
- 识别为：`description: "讨论 [[contract.docx]] 这个文件的内容"`
- 理由：
  - 文字更连贯
  - 明确标注附件位置
- 缺点：
  - 需要额外的标记语法
  - 增加解析复杂度

**方案 C: 智能替换**
- 识别为：`description: "讨论合同文件这个文件的内容"`
- 理由：
  - 文字最连贯
- 缺点：
  - 可能误判（无法确定是"合同文件"还是"协议文件"）
  - 增加处理逻辑

**📌 推荐：方案 A（保持现状）**
- 简单直接
- 大模型可以结合 `attachments` 和 `rawDescription` 理解
- 实际使用中，用户更可能将路径单独成行

### 问题 2: 路径后的文字

**场景**:

**识别结果**:
```json

  "description": "这个文件很重要",
  "attachments": [{ "path": "C:\\docs\\file.docx" }]
```

**分析**:
- ✅ 这是**符合预期**的
- ✅ 文字描述清晰（"这个文件很重要"）
- ✅ 附件路径明确（单独列出）
- ✅ 大模型可以理解"这个文件"指的是附件中的 docx

**📌 推荐：保持当前逻辑**
- 符合设计目标（文字和附件分离）
- 大模型可以结合上下文理解
- 用户习惯后，自然会调整输入方式（如路径单独成行）

### 问题 3: 多个连续空行

**场景**:
```
讨论合同条款

C:\docs\contract.docx

这次会议很重要
```

**分析**:
- 当前逻辑：`text.replace(/\n\s*\n/g, '\n')` 会将多个空行压缩为一个
- 结果：
  ```
  讨论合同条款
  这次会议很重要
  ```

**建议方案**:

**方案 A: 保留原始格式（推荐）**
- 不压缩空行，保留用户输入的格式
- 理由：
  - 空行可能是用户有意为之（分段）
  - 保持原文的可读性
  - 大模型可以理解段落结构
- 实现：只移除路径，不处理空行

**方案 B: 压缩空行**
- 将多个连续空行压缩为一个
- 理由：
  - 减少冗余空白
  - 使文本更紧凑
- 缺点：
  - 可能丢失用户的格式意图

**📌 推荐：方案 A（保留原始格式）**
- 更尊重用户的输入习惯
- 段落结构更清晰
- 对大模型分析更友好

### 问题 4: 路径的识别优先级（新增）

**场景**:
```
查看 https://example.com/doc 和 C:\docs\file.docx
```

**分析**:
- URL 和 Windows 路径都应该被正确识别
- 当前正则表达式按顺序匹配，不会冲突

**📌 推荐：保持当前逻辑**
- 先匹配 URL，再匹配 Windows 路径
- 两者不会冲突

## 最终解析逻辑

```typescript
function extractAttachments(description: string): {
  text: string
  attachments: Array<{
    path: string
    name: string
    type: AttachmentType
  }>
} {
  const attachments: Array<{
    path: string
    name: string
    type: AttachmentType
  }> = []
  
  let text = description
  
  // 正则表达式匹配各种路径（按优先级排序）
  const patterns = [
    // URL 链接
    /(https?:\/\/[^\s<>"{}|\\^\`\[\]]+)/g,
    // Windows 路径（反斜杠）
    /([A-Za-z]:\\[^\s<>"|?*]+)/g,
    // Windows 路径（正斜杠）
    /([A-Za-z]:\/[^\s<>"|?*]+)/g,
    // UNC 路径
    /(\\\\[^\s<>"|?*]+)/g,
    // 相对路径
    /(\.?[\/\\][^\s<>"|?*]+\.[^\s<>"|?*]+)/g
  ]
  
  for (const pattern of patterns) {
    const matches = description.match(pattern) || []
    for (const match of matches) {
      // 提取文件名
      const name = match.split(/[\/\\]/).pop() || ''
      
      // 判断类型
      const type = getAttachmentType(match)
      
      attachments.push({
        path: match,
        name,
        type
      })
      
      // 从文本中移除路径（替换为空）
      text = text.replace(match, '')
    }
  }
  
  // 清理多余空白（但保留段落结构）
  text = text.replace(/  +/g, ' ').trim()
  
  return { text, attachments }
}
```
